16,701 availability findings. Zero crashes.

Basquin was pointed at two real, unmodified JVM web apps — Apache JSPWiki and MyBatis JPetStore — for ~10 minutes each. It found 16,701 moments where an app degraded: allocating megabytes on a single page render, stalling for seconds. Every request returned a normal 2xx/3xx. A crash-only tool — a fuzzer, a load test's error count — reports none of these.

The cold cliff

8260 ms → 847 ms → 34 ms

JSPWiki's front page on the first three hits after deploy. The first-request-after-deploy cliff — JSP compilation plus cold-cache markup rendering — is an availability pathology a warmed-up load test never sees and a crash test never flags. Basquin catches it on request #1.

What a load tool sees vs what Basquin adds

k6 and Basquin measure different things, and that's the point. On the same JPetStore workload:

*Indicative only, not a controlled head-to-head. The ~256 ms figure reflects the valve serializing requests (a deliberate measurement tradeoff — DD-005 — so per-request heap/thread deltas stay accurate under concurrency); single-flight latency is ~26 ms. The point isn't speed — it's that a load tool reports zero defects on requests Basquin flags 2,347 times.

Methodology & honesty

  • Real runs only. Every figure traces to a file under bench-results/ (sampler CSVs, server-finding logs, k6 JSON) — nothing projected or hand-tuned.
  • Apps unmodified. Apache JSPWiki 2.12.4 (prebuilt WAR) and MyBatis JPetStore-6 (built at pinned commit c8f62cc7), each on Tomcat 9 with the namespace-free Basquin valve + agent. No code, config, or bytecode changes to the apps.
  • Budgets: soft mode, latency>25ms, heapDelta>256KB — findings are recorded, the run continues.
  • Driver: HttpRouteDriveTarget, single-flight, ~20 req/s, 12,000 requests/app across a fixed route mix. Client latency from k6 at 1 VU.
  • Known gap: this pass ran the round-robin/load driver. Coverage-guided exploration (guided-vs-naive coverage climb) needs a JaCoCo agent in each app JVM and is a documented next addition — it would raise the finding counts, not lower them.
  • k6 comparison is explicitly indicative, not a controlled benchmark; the tools measure different things and the valve's serialization is noted above.
