# Apache Roller 6.1.5 — bench target #3 (the DB-backed one)

Roller is the target that makes the bench honest about **networked databases**. JPetStore talks to an
in-process HSQLDB and JSPWiki to a directory of text files; Roller talks JPA/EclipseLink to a
PostgreSQL over the network, renders through Velocity, and indexes through Lucene. It is also the
only one of the three with a real login, a CSRF-guarded authoring UI, and an unauthenticated write
path (comments) — so it is where read/write interleaving, cache invalidation and connection-pool
behaviour actually show up.

It was previously abandoned ("Roller — stretch (DB-backed) — attempted thoroughly, OUT" in
[`../README.md`](../README.md)) on a context-listener failure. That failure is
[diagnosed and fixed below](#the-one-that-sank-the-last-attempt-javaxmailsession); Roller is in.

| | |
|---|---|
| Version | Apache Roller **6.1.5** (binary dist, sha256 `c0f16e07…07ea8`) |
| Servlet namespace | `javax` (web.xml 4.0) → **Tomcat 9** + the namespace-free valve |
| Storage | **PostgreSQL 16** over the network (Deployment + Service), Lucene index on local disk |
| Images | `basquin/roller-raw:0.2.0` (app), `basquin/roller-db:0.2.0` (seeded DB) |

## Build, deploy, run

```bash
cd deploy/bench/roller
bash setup.sh                                   # fetch + verify dist, explode WAR, emit seed/
docker build -f Dockerfile.db  -t basquin/roller-db:0.2.0  .
docker build -f Dockerfile.raw -t basquin/roller-raw:0.2.0 .
kind load docker-image basquin/roller-db:0.2.0  --name basquin
kind load docker-image basquin/roller-raw:0.2.0 --name basquin

kubectl apply -f k8s.yaml                       # DB + app + Services, already installed
kubectl -n basquin-system rollout status deploy/roller
bash smoke.sh                                   # proves it is SEEDED, not just answering 200

kubectl apply -f basquin.yaml                   # BasquinTarget + a 30m explore BasquinCampaign
```

The campaign needs its two ConfigMaps (ONBOARDING.md §5) — from the repo root:

```bash
kubectl -n basquin-system create configmap roller-grammar \
  --from-file=roller.grammar=examples/grammar/roller.grammar
kubectl -n basquin-system create configmap roller-corpus \
  --from-file=examples/corpus/roller/ --from-file=examples/corpus/roller/values/
```

Laptop instead of cluster: `TOMCAT_HOST_PORT=8091 docker compose up -d`, then
`bash probe.sh http://localhost:8091`.

Idle it between benchmark arms (single-node kind clusters are shared):
`kubectl -n basquin-system scale deploy/roller deploy/roller-db --replicas=0`.

## Why a seeded DB image

Roller ships a first-run **install wizard**: connect to an empty database, click "create tables",
register the first user, create a weblog. None of that is scriptable in a `kubectl apply`, and a
target that needs six manual clicks before it serves anything is not a reproducible benchmark.

Options considered:

| Option | Verdict |
|---|---|
| Embedded Derby/HSQLDB baked into the app image | One container, but the seed can only be produced by *running* Roller and copying its data dir — an opaque binary blob in git-ignored land, and it drops the networked-DB dimension that is the reason Roller is here at all. |
| Postgres + a seed `Job` that runs after the app installs itself | Needs ordering between two pods and an app that is briefly serving a wizard. Fragile, and "briefly wrong" is exactly the state a fuzzer will catch you in. |
| Postgres + a **hand-clicked** install, `pg_dump`ed | Reproducible only if you keep the dump; the dump is then a 200KB opaque artifact nobody can review or regenerate. |
| **Postgres image with the schema + a generated seed in `/docker-entrypoint-initdb.d/`** ✅ | `postgres:16` runs those scripts once, in filename order, when the data dir is empty. The app pod finds a fully installed database on its first connection, so there is no wizard state to race. The schema is **Roller's own `dbscripts/postgresql/createdb.sql`, copied verbatim from the dist**, and the data is ~200 lines of plain INSERTs generated by `setup.sh` — reviewable, diffable, regenerable. |

The seed is one user (`basquin` / `basquin`, global `admin`+`editor`), one weblog (`basquin`,
theme `basic`), 5 categories, 40 published entries dated one day apart, 80 comments, 40 tags.
Every id is fixed, not random, so the corpus values in `examples/corpus/roller/values/` keep pointing
at real rows across a rebuild.

**What the seed does not have to do:** build the Lucene index. Roller notices the index is
inconsistent with the database on startup and rebuilds it in the background (~0.3s for 40 entries),
so `/basquin/search?q=…` returns real hits from a purely SQL seed.

## Gotchas

### The one that sank the last attempt: `javax.mail.Session`

Symptom: `SEVERE ... One or more listeners failed to start`, `Context [] startup failed`, and then —
the trap — `INFO ... Server startup in [2900] milliseconds`. Tomcat looks healthy, the container is
"up", the readiness probe on `/` gets a 404, and nothing in `catalina.out` says why. The actual
exception is in `logs/localhost.<date>.log`:

```
Caused by: java.lang.ClassNotFoundException: javax.mail.Session
```

The Roller binary dist ships **no JavaMail** — its install guide assumes you drop `mail.jar` into the
*server's* lib, and `javax.activation` left the JDK after 11. `RollerContext`'s listener builds a
`MailProvider` unconditionally, so with either jar missing the whole context dies. `setup.sh` fetches
both into `tomcat-lib/` and `Dockerfile.raw` copies them into `/usr/local/tomcat/lib` — the server's
classpath, so the deployed webapp stays byte-identical to the upstream WAR.

**Three logs, not one.** Roller's own log4j2 writes to `logs/roller.log`, the listener stack trace
goes to `logs/localhost.<date>.log`, and only the summary reaches stdout. `docker logs` alone will
never tell you why Roller failed.

### A seeded database still shows the install wizard

`DatabaseInstaller` decides what to do from `roller_properties['roller.database.version']`. Tables
present + version row absent reads as "ancient schema", and **every** route renders *"Database tables
need to be upgraded"* — with a 200 status, so probes and smoke tests pass. The value is Roller's own
`parseVersionString`: strip non-digits from `ro.version` and keep three (`6.1.5` → `615`).
`setup.sh` derives it from the dist rather than hardcoding it.

### The `{bcrypt}` prefix on the password is load-bearing

`RollerContext` builds a Spring Security `DelegatingPasswordEncoder`, and because
`passwds.encryption.lazyUpgradeFrom=SHA`, an **un-prefixed** hash is interpreted as a legacy SHA-1
digest. A perfectly valid bare `$2a$10$…` therefore never matches: login just bounces to
`/roller-ui/login.rol?error=true`, with nothing in any log. Store `{bcrypt}$2a$10$…`.

### EclipseLink caches the user row

If you patch `roller_user` with `psql` on a running Roller, the L2 cache serves the stale row and
login keeps failing after you have fixed it. Restart the app pod. (Irrelevant in the real flow — the
seed is loaded before Roller ever connects — but it will waste twenty minutes if you are debugging.)

### The comment form is guarded by a CAPTCHA the theme never renders

Roller's default `comment.authenticator.classname` is `MathCommentAuthenticator`: it puts *"what is
3 + 5?"* in the session and requires an `answer` parameter. None of the bundled themes render the
question, so the app's only unauthenticated write path answers *"The response is not correct"* to
**every** client, browser included. `setup.sh` configures `DefaultCommentAuthenticator` in
`roller-custom.properties` — deployment config, in the same spirit as JSPWiki's
`jspwiki-custom.properties`; the WAR is untouched.

### Routes that are not what the docs suggest

- The comment POST goes to **the entry permalink itself** (`POST /basquin/entry/<anchor>` with
  `method=post`), not to `/roller-ui/rendering/comment/*` — which is the mapping in `web.xml`.
- Date archives are parsed **by length**: `/date/202607` (month) and `/date/20260701` (day) work,
  `/date/2026` is a 404.
- `/basquin/tags/<tag>` 404s on the bundled themes even with tags and tag aggregates seeded; the
  `?tags=` query form is accepted but does not filter. Tags are not in the grammar for that reason.
- `/basquin/page/<name>` only resolves *custom* templates, not the theme's built-ins.

### Build/deploy hygiene

- Exploded `ROOT/`, never `ROOT.war` — the coverage initContainer copies class files out of the image.
- Roller 6.1.5 **does** ship real classes in `WEB-INF/classes` (584 of them, unlike JSPWiki's
  jar-only layout), so `classesPath` would work there. `Dockerfile.raw` still stages just
  `org/apache/roller` into `/basquin-app-classes`, because `WEB-INF/classes` also holds tens of MB of
  themes, i18n bundles and dbscripts that the initContainer would copy for nothing.
- `-Xmx2g`. The ONBOARDING gotcha applies unchanged.
- The app Deployment has an **initContainer that waits for the DB**. Without it, an app pod that wins
  the race against Postgres starts, fails to bootstrap, and then serves 404s from a "Ready" pod
  forever — a `CrashLoopBackOff` would at least be honest.
- The DB Service **must** be named `roller-db`: that hostname is compiled into the JDBC URL in
  `roller-custom.properties`.
- Don't bind-mount `webapp/` from a Windows drive under WSL. Tomcat scans 109 jars at deploy; over
  `/mnt/c` that took >6 minutes and looked like a hang. From an image layer it is ~4 seconds.

## Grammar

[`examples/grammar/roller.grammar`](../../../examples/grammar/roller.grammar). Aimed at the measured
cost distribution (`probe.sh`, warm):

| Route | Cost | Why it is in the grammar |
|---|---|---|
| `/basquin/search?q=…` | **~530ms** | Lucene query + `searchresults.vm`, uncached — 10-25× everything else |
| `/basquin/date/YYYYMM` | ~50ms | month archive loops per-day, one query each |
| `/basquin/feed/entries/{rss,atom}` | ~40ms | ROME serialization |
| `/basquin/entry/<anchor>` | ~20ms | permalink + comment list |
| `/basquin/`, `/` | ~2ms | cache hits — in the grammar because the writes below invalidate them |

Writes:

- **`POST /basquin/entry/<anchor>`** (unauthenticated comment) — works, and uses DD-038 `<nonce>` so
  every replayed comment is a distinct row that invalidates the entry's cached render. Verified: a
  3-minute explore wrote **16 comments** with distinct nonces into `roller_comment`.
- **`@sequence login_publish`** (authenticated entry publish, DD-036 `<<salt=input:salt`) — the salt
  capture is correct and the flow works by hand, but the step is **skipped at runtime** for a reason
  that is a Basquin limitation, not a grammar bug. See below.

Roller's CSRF token is not `struts.token`: it is a per-session one-shot **`salt`**
(`SaltCache`, `<s:hidden name="salt"/>`) validated by `ValidateSaltInterceptor`. Static field name,
so DD-036's `input:` capture is sufficient — no DD-037 `inputpair` needed.

## What Basquin can't express (yet)

Found by trying to reach Roller's authoring surface. Both are correlation gaps, not grammar gaps.

1. **A login that rotates the session cookie is uncorrelatable.** Spring Security's form login
   answers `302` **and issues a new `JSESSIONID` on that 302** (session-fixation protection). The
   runner sets `setInstanceFollowRedirects(true)` and scrapes `Set-Cookie` off the *final* response,
   so the rotated cookie is silently dropped and every later step in the sequence is anonymous. The
   downstream symptom is honest but misleading: `unresolved correlation ref … step skipped`, because
   the anonymous `entryAdd.rol` 302s to the login page and has no `salt` input to capture. Verified
   against the live app: `JSESSIONID 5E9F79…` before login, `ADA9D5…` on the 302 after.
   *Fix shape:* capture `Set-Cookie` from intermediate responses (either follow redirects manually,
   or capture before following). This unlocks **every** Spring Security / form-login app, not just
   Roller — probably the highest-value correlation feature left.
2. **No way to send request headers from a grammar.** Roller's AtomPub endpoint
   (`/roller-services/app/*`) is a complete authenticated write API guarded by HTTP **Basic** auth —
   no session, no CSRF, ideal for a fuzz corpus — and it is unreachable because a route template can
   express a method, a path and a body, but not an `Authorization` header. A `>>Header: value`
   request-side directive (the mirror of `<<name=header:`) would open it, plus `Accept`/`Content-Type`
   negotiation surfaces generally.
3. **Captured values can only be replayed, never computed.** Roller's default comment CAPTCHA wants
   the *sum* of two numbers rendered into the page. Capture-and-substitute cannot add. This one is
   arguably out of scope by design (DD-036 rejected "magic" for good reasons) — noted because it is
   the reason the deployment has to swap the authenticator, and because any app with a
   challenge-response step will hit the same wall.
