Getting started

Clone, build, catch a deliberate leak, then drive a real app and read the live status panel. Every command below is a real Gradle task or flag.

Java 17+ (tested 17 & 21) Gradle wrapper included

1  Clone & build

Java 17 or newer is the only prerequisite; the Gradle wrapper is bundled.

# clone
git clone https://github.com/ianp94/basquin.git
cd basquin

# compile + run checks
./gradlew build

# build the fat jar (agent + runner + examples)
./gradlew jar   # -> build/libs/basquin-0.2.0.jar

2  See a leak get caught

The fastest way to see the point of the tool: run the deliberate thread-leak demo. It fails on purpose — a worker thread survives the iteration boundary and the harness reports it with a stack.

# thread-leak demo — non-zero exit, that's the demo working
./gradlew runRunnerLeak

# prove stability instead: 10,000 clean iterations, prints metrics
./gradlew runSoakProper
tip

For demos and CI, add -Dbasquin.forceExitOnLeak=true so the process exits immediately on a leak or hard invariant instead of finishing the run.

3  Drive a running web app

Point the HTTP driver at any running app to measure real end-to-end latency, treat 5xx as a crash, and watch the live status screen. If the app carries the Basquin valve/filter, the driver also harvests server-side X-Basquin-Invariant-* response headers.

./gradlew runHttpDrive \
  -Dexamples.http.baseUrl=http://localhost:8080 \
  -Dbasquin.invariant.latency.maxMs=50 \
  -Dbasquin.invariant.mode=soft \
  -Ddrive.iterations=200

Set a latency budget with -Dbasquin.invariant.latency.maxMs and use soft mode to record and keep going (rather than hard, which fails fast on the first violation). For coverage-guided exploration with a request grammar and a live dashboard, see the Kubernetes / Tomcat guide.

4  Run against embedded Tomcat

Basquin ships a small embedded-Tomcat target with three routes — /crash, /latency, /heap — that exercise both the crash and the invariant oracles inside a single JVM (opt-in; JQF is not run in CI).

# coverage-guided exploration of the embedded Tomcat routes
./gradlew runFuzzTomcatJQF -DenableJQF=true

# deterministic replay of the saved corpus (no JQF needed)
./gradlew runTomcatCorpus

Saved inputs land under fuzz-results/<target>/ as a .bin plus a .meta.txt triage report (classification, timestamp, invariant violated, stacks).

5  Point it at your app

Implement the three-method IterationTarget interface — one iteration is one request or unit of work. Optionally implement InputReceiver to accept fuzz input.

package your.pkg;
import runner.api.IterationTarget;

public class YourTarget implements IterationTarget {
  public void initialize()       throws Exception { /* start app/server */ }
  public void executeIteration() throws Exception { /* one request / unit of work */ }
  public void close()            throws Exception { /* cleanup */ }
}
java -cp build/libs/basquin-0.2.0.jar:<your-cp> \
  runner.GenericRunner 100 your.pkg.YourTarget

Key flags

The flags you'll reach for first. The full catalog is in USAGE.md.

Invariants

Disabled unless set. Modes: hard (fail fast) or soft (record and continue); mode can be overridden per invariant (e.g. ...latency.mode).

FlagMeaning
-Dbasquin.invariant.latency.maxMs=100Latency budget per iteration
-Dbasquin.invariant.heapDelta.maxKb=256Max heap growth per iteration
-Dbasquin.invariant.threadDelta.max=2Max surviving new threads
-Dbasquin.invariant.mode=hard|softFail fast, or record + continue
-Dbasquin.invariant.stack=current|all|offEvidence captured on violation (default current)
-Dbasquin.invariant.latency.sample=trueSample the execution stack at the latency threshold

Runner & measurement

FlagMeaning
-Dbasquin.status=trueLive AFL-style status screen (suppresses per-iteration log spam)
-Dbasquin.status.forceTty=trueRender the status box even when output is piped
-Dbasquin.heap.gcBeforeMeasure=trueGC before heap baseline so the delta is retention, not allocation noise
-Dbasquin.forceExitOnLeak=trueExit immediately on a leak / hard invariant (demos, CI)
-Dbasquin.native.lib=<abs path>Load the JVMTI native agent (same path as -agentpath)

HTTP driver

FlagMeaning
-Dexamples.http.baseUrl=<url>Target base URL for the HTTP driver
-Dexamples.http.routes=<csv>Routes to drive
-Ddrive.iterations=<n>Number of requests to send

Reading the status panel

With -Dbasquin.status=true the runner draws a live AFL-style status screen (redraw interval -Dbasquin.status.intervalMs, default 1000ms). What the rows mean:

RowWhat it tells you
execs / execs/secIterations completed and throughput.
crashesGenuine faults only — unhandled exceptions and 5xx. Expected input rejections are counted separately as rejected, not here.
invariant findsIterations that tripped a latency / heap / thread invariant. On a valve-instrumented app these are harvested server-side.
rejectedInputs a target's CrashClassifier declared as expected validation failures — not bugs.
corpus / last findCorpus size and time since the last interesting input, during exploration.
coverage %Real "% of code explored" in the app under test, read from its JaCoCo agent — lights up only when a coverage source reports.
